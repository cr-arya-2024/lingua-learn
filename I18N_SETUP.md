# Lingo.dev i18n Integration Guide

## Overview

This project uses **Lingo.dev CLI** for automated multilingual translation management. The app supports:
- **Source Locale**: English (en)
- **Target Locales**: Spanish (es), French (fr), Hindi (hi)

## Project Structure

```
lingua-learn/
├── i18n.json                    # Lingo.dev configuration
├── locales/
│   ├── en.json                 # English source strings
│   ├── es.json                 # Spanish translations (auto-generated)
│   ├── fr.json                 # French translations (auto-generated)
│   └── hi.json                 # Hindi translations (auto-generated)
├── lib/
│   ├── i18n.ts                 # Translation helper functions
│   └── locale-context.tsx      # React locale context
├── middleware.ts               # Locale detection middleware
├── app/
│   └── [locale]/
│       ├── layout.tsx          # Localized layout wrapper
│       ├── page.tsx            # Example localized page
│       └── ...                 # Other localized routes
└── package.json                # NPM scripts including i18n:translate
```

## Setup Instructions

### 1. Install Dependencies

```bash
npm install
```

### 2. Environment Variables

Set up your Lingo.dev API key:

```bash
# .env.local
LINGO_API_KEY=your_lingo_dev_api_key_here
```

Get your API key from [Lingo.dev Dashboard](https://lingo.dev/dashboard).

### 3. Initialize Translations

The base English translations are already in `locales/en.json`. To generate translations for es, fr, and hi:

```bash
npm run i18n:translate
```

This command:
- Reads `i18n.json` configuration
- Processes `locales/[locale].json` files
- Uses Lingo.dev engine to generate translations
- Creates/updates translated JSON files
- Generates `i18n.lock` file (commit this to version control)

### 4. What Gets Generated

After running `npm run i18n:translate`, you'll see:

```
locales/
├── en.json      # Original (unchanged)
├── es.json      # Spanish (generated by Lingo.dev)
├── fr.json      # French (generated by Lingo.dev)
├── hi.json      # Hindi (generated by Lingo.dev)
└── i18n.lock    # Lock file for translation consistency
```

**All translation files and `i18n.lock` should be committed to git.**

## Using Translations in Code

### Server-Side (App Router)

```typescript
import { t, Locale } from '@/lib/i18n';

export default function Page({ params }: { params: { locale: string } }) {
  const locale: Locale = params.locale as Locale;
  
  return (
    <div>
      <h1>{t('landing.title', locale)}</h1>
      <p>{t('landing.subtitle', locale)}</p>
    </div>
  );
}
```

### Client-Side

```typescript
'use client';

import { useLocale } from '@/lib/locale-context';
import { t } from '@/lib/i18n';

export function MyComponent() {
  const { locale, setLocale, supportedLocales } = useLocale();
  
  return (
    <div>
      <h1>{t('landing.title', locale)}</h1>
      
      {/* Locale Switcher */}
      {supportedLocales.map((loc) => (
        <button key={loc} onClick={() => setLocale(loc)}>
          {loc.toUpperCase()}
        </button>
      ))}
    </div>
  );
}
```

## Configuration Details

### i18n.json

The Lingo.dev configuration file specifies:
- **Buckets**: Define which files to translate
- **Source Locale**: English (en)
- **Target Locales**: es, fr, hi
- **Engine**: Uses Lingo.dev engine for translations

```json
{
  "buckets": [
    {
      "id": "lingua_learn_strings",
      "include": ["locales/[locale].json"],
      "sourceLocale": "en",
      "targetLocales": ["es", "fr", "hi"]
    }
  ],
  "engine": "lingo.dev",
  "engines": {
    "lingo.dev": {
      "apiKey": "${LINGO_API_KEY}"
    }
  }
}
```

## Adding New Translations

1. **Edit `locales/en.json`** to add new strings:

```json
{
  "new_feature": {
    "title": "My New Feature",
    "description": "This is a new feature"
  }
}
```

2. **Run the translation command**:

```bash
npm run i18n:translate
```

3. **Commit** the updated JSON and lock files:

```bash
git add locales/ i18n.lock
git commit -m "Add new translations for feature X"
```

## Locale Routing

The app uses dynamic routes with locale parameters:

- `/en` → English version
- `/es` → Spanish version
- `/fr` → French version
- `/hi` → Hindi version

**Middleware automatically**:
- Detects user's preferred language (cookie or accept-language header)
- Redirects to appropriate localized URL
- Sets locale context for the session

## Translation Fallback

The `t()` function implements smart fallback:
1. Looks for translation in requested locale
2. Falls back to English if translation is missing
3. Returns the key itself as last resort

```typescript
t('key.that.does.not.exist', 'es')
// Returns: 'key' (in Spanish if available, else English, else key name)
```

## Best Practices

✅ **DO:**
- Keep translations in `locales/*.json` only
- Use dot notation for nested keys: `"section.key"`
- Commit `i18n.lock` to ensure consistency
- Run `i18n:translate` before major commits

❌ **DON'T:**
- Hardcode strings in components (use `t()` instead)
- Edit translated files manually (let Lingo.dev handle it)
- Forget to run `i18n:translate` after adding new strings

## Troubleshooting

### "Command not found: lingo.dev"
Reinstall dependencies:
```bash
npm install
```

### "No translations generated"
Check:
- `LINGO_API_KEY` is set in `.env.local`
- `i18n.json` exists at project root
- `locales/en.json` has content

### "Translations not updating on page"
- Clear `.next` folder: `rm -rf .next`
- Restart dev server: `npm run dev`
- Ensure middleware is active: check `middleware.ts` exists

## CI/CD Integration

Add to your CI pipeline:

```bash
npm run i18n:translate
git diff --exit-code locales/ i18n.lock || exit 1
```

This ensures translations are always up-to-date before merging.
